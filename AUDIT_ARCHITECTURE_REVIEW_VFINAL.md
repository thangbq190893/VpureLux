# Audit Architecture Review vFinal

Status: Final architecture review awaiting implementation approval

## 1. Executive Decision

The Audit bounded context will provide an immutable, append-only business audit
ledger named `BusinessAuditLog`.

ABP Audit Logging remains enabled and continues to own technical auditing:

- HTTP request and application-service execution records.
- Entity change tracking generated by ABP/EF Core.
- Execution duration, client information, exceptions, and technical actions.
- Existing ABP audit-log administration UI and APIs.

The Audit bounded context owns business-semantic auditing:

- Critical committed business actions.
- Stable business entity identifiers and human-readable references.
- Curated Before/After change sets.
- Actor, correlation, causation, module, action, and event metadata.
- Cross-module business timelines and audit reports.

Business Audit does not replace ABP Audit Logging, duplicate every ABP action,
or become an event-sourcing system. Business entities remain the source of
truth. Audit records are evidence and reporting facts only.

## 2. Documentation Reconciliation

This review reconciles the certified Catalog, BOM, Customer, Pricing,
Inventory, and Sales modules with:

- ADR-008: Full Audit Logging, ABP Audit Logs, and an additional
  `BusinessAuditLog` table.
- ADR-009: ERP historical integrity and soft-delete strategy.
- ADR-010: Domain Events for loose coupling.
- Module Specification rules `AUDIT-001` through `AUDIT-003`.
- Database Schema Specification `BusinessAuditLogs`.
- API Contract Specification `/api/audit/logs`.
- Reporting Specification audit reports and CQRS read-side allowance.
- System master rule that audit records must never be deleted.

### Current Repository State

- ABP Audit Logging modules are registered in Domain, Application, EF Core,
  and Web.
- `app.UseAuditing()` is enabled after authorization and Unit of Work.
- Existing ABP tables include `AbpAuditLogs`, `AbpAuditLogActions`,
  `AbpEntityChanges`, and `AbpEntityPropertyChanges`.
- Existing domain events cover the major certified-module state changes.
- Catalog currently writes business comments into the active ABP audit log.
  This is useful technical context but is not a durable business-audit model.

### Required Alignment During Audit Implementation

- Replace or retire `CatalogBusinessAuditEventHandler` after equivalent Catalog
  events are captured by the centralized Business Audit handlers, preventing
  duplicate business records.
- Add narrowly scoped audit event metadata for update workflows where current
  lightweight domain events cannot provide a correct Before/After change set.
- Do not change certified aggregate boundaries, business invariants, state
  machines, or module ownership.

## 3. Final Audit Ownership

### Audit Owns

- `BusinessAuditLog` append-only records.
- Audit event classification and schema versioning.
- Safe serialization, masking, and snapshot policies.
- Correlation-based and entity-based audit queries.
- User activity, entity change, price change, BOM change, inventory action,
  and sales action reports.
- Audit export orchestration.
- Audit retention and immutability enforcement.

### Audit Does Not Own

- Current business entity state.
- Inventory ledger facts or inventory balances.
- Sales snapshots, revenue, profit, or customer purchase history.
- Pricing history or BOM history.
- Customer or Catalog master data.
- Authentication, authorization, or identity lifecycle.
- ABP technical audit records.
- Recovery, rollback, or reversal of business operations.

## 4. Relationship With ABP Audit Logging

| Concern | ABP Audit Logging | Business Audit |
|---|---|---|
| Primary purpose | Technical traceability | Business traceability |
| Source | Middleware, interceptors, EF changes | Committed domain/business events |
| Failed requests | Yes | No, unless a future explicit rejected-action policy is approved |
| Exceptions | Yes | No |
| Entity property changes | Generic technical changes | Curated business-safe change set |
| Business action names | Not guaranteed | Required |
| Cross-module timeline | Correlation query possible | First-class query |
| Retention | Existing ABP records | Immutable forever |
| Public mutation API | None | None |

Business Audit records will store the ABP `CorrelationId`. An optional
`TechnicalAuditLogId` may be stored when it is reliably available, but there
will be no database foreign key to ABP audit tables. Correlation remains the
stable integration mechanism and avoids lifecycle coupling.

Business Audit handlers run inside the same Unit of Work as the committed
business action. A failed or rolled-back business operation must not produce a
successful Business Audit record. ABP Audit Logging remains responsible for
recording the failed technical execution.

## 5. Business Audit Scope

### Mandatory Events

#### Catalog

- Component created, updated, activated, deactivated.
- Product created, updated, activated, deactivated.
- Future image added, changed, or removed using metadata and `ImageHash` only.
  Base64 image content must never be stored in audit payloads.

#### BOM

- BOM version created.
- BOM draft structure changed.
- BOM version cloned.
- BOM published.
- BOM archived.

Published and archived BOM records are immutable. Audit records describe the
state transition and version identity; they do not duplicate the complete BOM
history already owned by BOM.

#### Customer

- Customer created and updated.
- Customer assigned to another Customer Group.
- Customer activated and deactivated.
- Customer Group created, updated, activated, and deactivated.

Sensitive customer fields must follow masking rules. Audit reports must not
become an unrestricted customer-data export.

#### Pricing

- Component Purchase Price Version created and closed.
- Product Suggested Selling Price Version created and closed.

Price audit records include price type, item identifier, version number,
effective period, VND amount, and Reason. Pricing remains the source of truth
for price history.

#### Inventory

- StockItem created, activated, and deactivated.
- Warehouse created, updated, activated, and deactivated.
- Receipt posted.
- Issue posted.
- Adjustment posted, including mandatory adjustment Reason.
- Lot depleted as a system-generated informational event.

Audit records reference InventoryTransaction identifiers and summary values.
They do not duplicate lot-allocation details or replace the Inventory ledger.

#### Sales

- Sales Order created.
- Draft line added, updated, or removed.
- Actual Selling Price overridden, including OverrideReason.
- Sales Order confirmed.
- Sales Order cancelled.

Confirmation audit records include OrderNo, CustomerId, totals, and status
transition summary. Full Sales snapshots remain owned by Sales.

#### Audit Operations

- Audit export requested and completed.
- Audit configuration or retention-policy changes, if introduced later.

Export actions must themselves be audited through ABP technical auditing and a
Business Audit record.

### Events Not Stored As Business Audit Records

- Read-only queries and page views. ABP technical audit may record them.
- Routine validation failures and exceptions. ABP technical audit owns them.
- Rebuilds of read models that do not alter business facts.
- Duplicate copies of immutable business history already stored in owning
  modules.

## 6. Data and Snapshot Policy

### Required Stored Data

Every Business Audit record stores:

- Immutable event identifier.
- Tenant identifier when multi-tenancy is enabled.
- Module and event name.
- Business action.
- Primary entity type, identifier, and display reference.
- Occurrence timestamp in UTC.
- Actor type, UserId, and UserName snapshot.
- CorrelationId and optional CausationId.
- Optional technical audit-log identifier.
- Before and After JSON change sets when applicable.
- Safe metadata JSON.
- Event schema version and severity.

### Before/After Snapshots

Before/After values are required for business-significant updates and state
transitions. They are selective change sets, not full serialized aggregates.

| Action type | Before | After |
|---|---|---|
| Create | Null | Safe creation summary |
| Update | Changed fields only | Changed fields only |
| State transition | Previous state | New state |
| Immutable version creation | Null | Version summary |
| Posting/confirmation | Draft or unposted status summary | Posted or confirmed summary |
| Delete/deactivate | Active/current status | Deactivated/deleted status |

Before values must be captured before mutation. Audit handlers must not infer
Before values by querying the already-mutated database state.

Current lightweight domain events do not contain sufficient Before/After data
for every update operation. Audit implementation may add dedicated,
business-safe audit events or change-set metadata at application orchestration
boundaries. These additions must remain lightweight and must not include large
object graphs.

### Data Exclusion and Masking

Never store:

- Passwords, tokens, secrets, connection strings, or authentication material.
- Raw Base64 image content.
- Complete request/response bodies by default.
- Large aggregate graphs.
- Raw exception stack traces in Business Audit.

Customer contact details and other sensitive values must be masked or omitted
according to an explicit safe-field allowlist. ABP technical audit
configuration must also exclude sensitive request parameters.

## 7. Final Audit Aggregate Design

### BusinessAuditLog Aggregate Root

`BusinessAuditLog` is an immutable aggregate root representing one committed
business event.

Core fields:

- `Id`
- `EventId`
- `TenantId`
- `Module`
- `EventName`
- `EventVersion`
- `Action`
- `EntityName`
- `EntityId`
- `EntityDisplay`
- `ActorType`
- `UserId`
- `UserNameSnapshot`
- `OccurredAtUtc`
- `CorrelationId`
- `CausationId`
- `TechnicalAuditLogId`
- `OldValueJson`
- `NewValueJson`
- `MetadataJson`
- `Severity`
- `IsSystemGenerated`
- `CreationTime`

Invariants:

- Records are append-only.
- Records cannot be updated, soft-deleted, or hard-deleted.
- `EventId`, module, event name, action, entity name, entity identifier,
  occurrence time, and correlation ID are required.
- JSON payloads must pass safe serialization and size limits.
- UserId may be null only for System or Integration actors.
- Event schema version must be positive.

`BusinessAuditLog` exposes creation only. It has no update, delete, activate,
deactivate, or state-transition behavior. It publishes no domain event, which
prevents recursive audit generation.

### Domain Service

`BusinessAuditManager` creates validated records from a normalized
`BusinessAuditEnvelope`. It owns:

- Safe-field allowlist enforcement.
- Masking and payload-size validation.
- Actor and correlation validation.
- Event classification and schema version validation.

Audit event handlers adapt certified-module domain events into normalized
envelopes. They must not contain business rules owned by source modules.

## 8. Final Audit Database Design

### Table: AppBusinessAuditLogs

| Column | SQL Server type | Rules |
|---|---|---|
| Id | UNIQUEIDENTIFIER | Primary key |
| EventId | UNIQUEIDENTIFIER | Required, unique |
| TenantId | UNIQUEIDENTIFIER NULL | Multi-tenant filter |
| Module | NVARCHAR(50) | Required |
| EventName | NVARCHAR(150) | Required |
| EventVersion | INT | Required |
| Action | NVARCHAR(100) | Required |
| EntityName | NVARCHAR(100) | Required |
| EntityId | UNIQUEIDENTIFIER | Required |
| EntityDisplay | NVARCHAR(300) NULL | Safe display reference |
| ActorType | TINYINT | User/System/Integration |
| UserId | UNIQUEIDENTIFIER NULL | Actor identifier |
| UserNameSnapshot | NVARCHAR(256) NULL | Historical display |
| OccurredAtUtc | DATETIME2 | Required |
| CorrelationId | NVARCHAR(64) | Required |
| CausationId | NVARCHAR(64) NULL | Optional event chain |
| TechnicalAuditLogId | UNIQUEIDENTIFIER NULL | No FK |
| OldValueJson | NVARCHAR(MAX) NULL | Safe changed fields |
| NewValueJson | NVARCHAR(MAX) NULL | Safe changed fields |
| MetadataJson | NVARCHAR(MAX) NULL | Safe event metadata |
| Severity | TINYINT | Informational/Important/Critical |
| IsSystemGenerated | BIT | Required |
| CreationTime | DATETIME2 | Required |

The table deliberately has no `IsDeleted`, `LastModificationTime`,
`LastModifierId`, or concurrency-update fields because records cannot change.

### Required Indexes

- Unique `UX_BusinessAuditLogs_EventId`.
- `IX_BusinessAuditLogs_EntityName_EntityId_OccurredAtUtc`.
- `IX_BusinessAuditLogs_Module_EventName_OccurredAtUtc`.
- `IX_BusinessAuditLogs_UserId_OccurredAtUtc`.
- `IX_BusinessAuditLogs_CorrelationId`.
- `IX_BusinessAuditLogs_TenantId_OccurredAtUtc`.
- `IX_BusinessAuditLogs_Action_OccurredAtUtc`.

### Immutability Enforcement

- Repository contract exposes insert and query operations only.
- No update or delete application contracts or HTTP routes.
- EF configuration does not use soft delete.
- SQL Server permissions deny application-role `UPDATE` and `DELETE`.
- A database trigger rejects `UPDATE` and `DELETE` as defense in depth.
- Audit records are not cascade-deleted and have no required foreign keys to
  business tables.

### Idempotency

`EventId` is unique. Phase 1 local-event adapters generate one stable event ID
per audit envelope within the business Unit of Work. Future distributed-event
integration must carry a producer-generated EventId and use the same unique
constraint.

## 9. Retention Strategy

Business Audit records are retained forever. Phase 1 keeps them online in SQL
Server and does not implement purge, soft delete, or hard delete.

ABP technical audit records are also retained without deletion under the
current master rule. A future archival policy may be proposed only through a
separate architecture and governance decision. Any future archive must be
lossless, immutable, queryable, verifiable, and must not weaken the rule that
audit evidence cannot be deleted.

Operational requirements:

- Monitor table and index growth.
- Use date-aligned SQL Server partitioning when volume justifies it.
- Back up audit records under the same or stronger policy as transactional
  data.
- Restrict direct database access.
- Audit all export and administrative access.

## 10. Final Audit Query and Reporting Design

Audit reporting is a CQRS read side. Query services may use optimized
read-only EF projections or SQL, must not load aggregates, and contain no
business mutation logic.

### Core Queries

- Paged audit-log search.
- Audit-log details.
- Entity timeline by EntityName and EntityId.
- Correlation timeline across modules.
- User activity timeline.

Filters:

- Module, event name, action, entity name, entity ID.
- User ID and actor type.
- Correlation ID.
- Severity.
- From/to UTC timestamps.

Default sorting is `OccurredAtUtc DESC, Id DESC`. All list endpoints require
paging and bounded date ranges for large exports.

### Required Reports

- User Activity Report.
- Entity Change Report.
- Price Change Report.
- BOM Change Report.
- Inventory Posting and Adjustment Report.
- Sales Confirmation, Cancellation, and Price Override Report.
- Cross-module Correlation Report.

Reports reference owning-module identifiers and snapshots. They do not
recalculate inventory balances, pricing history, sales profit, or other
business facts.

### Export Strategy

Exports require `Audit.Export`, use bounded filters, and should execute as
background jobs for large datasets. Export files must be access-controlled,
time-limited, and audited. Phase 1 export formats are CSV and Excel.

## 11. Final Audit API Design

Business Audit API:

- `GET /api/audit/logs`
- `GET /api/audit/logs/{id}`
- `GET /api/audit/entities/{entityName}/{entityId}`
- `GET /api/audit/correlations/{correlationId}`
- `GET /api/audit/reports/user-activity`
- `GET /api/audit/reports/entity-changes`
- `GET /api/audit/reports/price-changes`
- `GET /api/audit/reports/bom-changes`
- `GET /api/audit/reports/inventory-actions`
- `GET /api/audit/reports/sales-actions`
- `POST /api/audit/exports`

There are no create, update, or delete public routes. Controllers delegate only
to Audit application/query services.

Existing ABP technical audit APIs and UI remain separate and continue to use
ABP permissions and contracts.

## 12. Final Audit Permission Matrix

| Permission | Capability |
|---|---|
| `Audit.View` | View business audit logs, details, timelines, and reports |
| `Audit.Export` | Export permitted business audit data; requires `Audit.View` |

Existing ABP technical audit-log permissions remain unchanged. Audit
application services must enforce permissions server-side; UI visibility alone
is insufficient.

## 13. Certified Module Integration

### Catalog

Audit consumes Catalog lifecycle events. Current ABP-comment handler is
superseded by centralized business audit capture. Image audits store file
metadata and SHA256 hash only.

### BOM

Audit consumes version creation, draft structure change, clone, publish, and
archive events. BOM remains the source of complete version history.

### Customer

Audit consumes Customer and CustomerGroup lifecycle events and group
reassignment. Sensitive values use a safe-field allowlist and masking.

### Pricing

Audit consumes version-created and version-closed events with safe price,
effective-period, Reason, and item metadata. Pricing remains the source of
historical lookup.

### Inventory

Audit consumes posted receipt, issue, adjustment, StockItem, Warehouse, and lot
depletion events. Inventory ledger and FIFO allocations remain the source of
truth.

### Sales

Audit consumes order creation, draft-line changes, price override,
confirmation, and cancellation events. Sales snapshots remain the source of
truth for historical revenue, cost, profit, and margin.

## 14. Final Audit Testing Strategy

### Domain Tests

- Required-field and event-schema validation.
- Safe serialization and masking.
- Payload-size limits.
- User/System/Integration actor rules.
- No mutation behavior exposed by `BusinessAuditLog`.

### Handler and Integration Tests

- Every mandatory certified-module event creates exactly one Business Audit
  record.
- Failed and rolled-back business operations create no successful Business
  Audit record.
- CorrelationId and actor metadata propagate correctly.
- System-generated events support null UserId.
- Duplicate EventId is rejected idempotently.
- No recursive audit records are generated.

### Persistence Tests

- Required indexes and unique EventId.
- No soft-delete column.
- Update and delete are rejected.
- No cascade deletion from business entities.
- JSON and string length mappings.
- Migration and model-snapshot consistency.

### Query and Reporting Tests

- Entity and correlation timelines.
- User activity, entity changes, price changes, BOM changes, Inventory actions,
  and Sales actions.
- Paging, bounded dates, stable sorting, and tenant isolation.
- Masked fields remain masked in details and exports.

### Permission and API Tests

- `Audit.View` and `Audit.Export` enforcement.
- All GET routes and export route.
- No create, update, or delete routes.
- Export access and export action auditing.

### Retention and Security Tests

- No application deletion path exists.
- Database update/delete protection is effective.
- Sensitive fields, secrets, image Base64, and large object graphs are absent.
- Audit records survive source-entity soft deletion.

## 15. Final Audit Certification Strategy

Audit can be certified only when:

1. ABP technical audit remains operational and is not duplicated.
2. Every mandatory business event is covered by an automated handler test.
3. Before/After change sets are correct, safe, and captured before mutation.
4. Business Audit records are append-only at application and database levels.
5. User, system actor, correlation, and causation metadata are verified.
6. Cross-module entity and correlation timelines are correct.
7. Retention and sensitive-data policies are enforced.
8. API, permission, reporting, migration, and full regression tests pass.
9. Existing certified module invariants and ownership boundaries remain
   unchanged.

## 16. Open Implementation Dependencies

These are implementation dependencies, not architecture blockers:

- Add business-safe audit change-set events for Catalog updates, BOM draft
  changes, Customer updates, Warehouse updates, Sales draft-line changes, and
  Sales price overrides.
- Enrich Pricing audit adapters with safe price and Reason metadata without
  placing large payloads in domain events.
- Decide the exact maximum size for each JSON audit payload during Audit
  implementation planning.
- Confirm SQL Server application-role permissions and trigger deployment
  strategy with operations.

## 17. Final Recommendation

Proceed with Audit implementation using a separate immutable
`AppBusinessAuditLogs` ledger integrated through local domain-event adapters.
Keep ABP Audit Logging unchanged for technical auditing. Implement the narrow
event-metadata dependencies listed above without redesigning or reopening the
certified modules.

Audit architecture is ready for implementation planning. No Audit production
code has been created by this review.
